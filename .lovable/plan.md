## 3. cut_type System Verification

### Edge Function Logic: ✅ CORRECT

**generate-storyboard/index.ts** (lines 297-346):
```typescript
const DEFAULT_CUT_TYPES: Record<SceneRole, CutType> = {
  hook: "hard",
  problem: "hard", 
  story_a: "continuity",
  reset: "hard",
  story_b: "continuity",
  cta: "hard",
  atmosphere: "hard",
  establish: "hard",
};

// Rule 1: First scene is ALWAYS hard (T2V)
if (i === 0) computedCutType = "hard";

// Rule 2: hook/cta/reset are ALWAYS hard
else if (role === "hook" || role === "cta" || role === "reset") 
  computedCutType = "hard";

// Rule 3: continuity only if previous role is eligible
else if (prevRole && !CONTINUITY_SOURCE_ROLES.includes(prevRole)) 
  computedCutType = "hard";
```

**continue-story-chain/index.ts** (lines 389-406):
```typescript
// THE KEY CONDITIONAL:
let cutType = nextScene.cut_type || "hard"; // Defaults to hard if missing

// Provider switch forces hard cut
if (prevProvider && prevProvider !== selectedProvider) {
  cutType = "hard";
}

// Only use I2V for continuity cuts
if (cutType === "continuity" && !isFirstScene && latestThumbnail) {
  startingFrameUrl = /* potentially resized */;
}
// Hard cuts: startingFrameUrl stays undefined (T2V)
```

### Log Evidence: ✅ Confirms Logic Works

```
[chain-continue] Scene 3 cut_type="hard" (provider switch luma→sora) → T2V
[chain-continue] Scene 4 cut_type="hard" (provider switch sora→runway) → T2V
```

### ❌ Problem: Storyboard Has No cut_type

Because the UI drops `cut_type`, the chain always falls back to:
```typescript

```

This means **ALL scenes default to hard cut**, which is safe but defeats the continuity system.

## Technical Summary

The cut_type architecture is **correctly implemented in the edge functions** but has a **critical UI persistence gap**. The `StoryBuilderPanel` drops `cut_type` when mapping scenes, causing all stories to have null cut_types in the database. The chain logic correctly defaults missing cut_types to "hard", making the system safe but unable to use continuity (I2V).

Additionally, thumbnail dimensions are not being stored, so the resize logic never triggers for Sora I2V scenes, causing repeated dimension mismatch failures.

**Priority Order:**
1. Fix UI to persist `cut_type` (immediate)
2. Fix thumbnail dimension persistence (immediate)
3. Add motif injection logging (diagnostic)
4. Add audit fields to video_jobs (observability)
